# Неделя 1 — База данных + контейнеры: развёрнутый план

Ниже — конкретные шаги на первой неделе: схема БД, Liquibase, Entity, репозитории **и** развёртывание всего этого в контейнерах (PostgreSQL + бэкенд). К концу недели 1 одна команда `docker-compose up` поднимает БД и бэкенд.

---

## 1. Проектирование схемы БД

- Зафиксировать список таблиц и связей между ними (уже есть в отчёте: users, companies, moysklad_integrations, telegram_integrations, top_n_reports, top_n_entries).
- Упрощённая схема (без лишнего для MVP):
  - **users** — кто заходит в приложение (id, email, created_at).
  - **companies** — компания, привязанная к пользователю (id, name, user_id, created_at).
  - **moysklad_integrations** — подключение МойСклад к компании (id, company_id, token_encrypted, status, created_at).
  - **telegram_integrations** — подключение Telegram-канала к компании (id, company_id, bot_token_encrypted, channel_chat_id, status, created_at).
  - **top_n_reports** — сформированный отчёт ТОП-N (id, company_id, period_start, period_end, status, created_at, published_at).
  - **top_n_entries** — строки рейтинга в отчёте (id, report_id, employee_name, employee_id_ms, revenue, margin, favorite_product, rank).
- Нарисовать схему (на листе, в draw.io или по SQL) и сверить с отчётом.

**Результат:** один файл или скрин с итоговой схемой (таблицы + связи).

---

## 2. Liquibase: миграции и создание таблиц

- Добавить в проект зависимость Liquibase (Spring Boot Starter уже тянет совместимую версию при `spring-boot-starter-data-jpa` + драйвер БД; при необходимости явно добавить `liquibase-core`).
- Создать каталог для чейнджлогов, например `src/main/resources/db/changelog/`.
- Главный файл — `db/changelog/db.changelog-master.xml`, в нём подключаются остальные изменения (например, `include file="changelog/001-create-tables.xml"`).
- Создать чейнджлог создания таблиц (например, `001-create-tables.xml`):
  - В правильном порядке: сначала `users`, потом `companies`, затем таблицы с внешними ключами (`moysklad_integrations`, `telegram_integrations`, `top_n_reports`, `top_n_entries`).
  - Использовать теги `<createTable>`, `<addPrimaryKey>`, `<addForeignKeyConstraint>` (или один общий `<sql>` с `CREATE TABLE ...` — как удобнее).
- В `application.yml` указать:
  - `spring.liquibase.change-log=classpath:db/changelog/db.changelog-master.xml`
  - И оставить `spring.jpa.hibernate.ddl-auto=validate` (или `none`), чтобы структуру править только через Liquibase, а не через Hibernate.

**Результат:** при старте приложения Liquibase накатывает чейнджлоги, в БД появляются все таблицы. Для следующих изменений схемы — новые файлы чейнджлога (002, 003, …) и include в master.

## 3. Подключение БД к приложению

- Создать или открыть проект на Spring Boot (Maven/Gradle).
- Добавить зависимости: Spring Data JPA, драйвер PostgreSQL (или H2 для локальных тестов).
- В `application.yml` / `application.properties` прописать:
  - URL БД (например, `jdbc:postgresql://localhost:5432/dashboard_battle`),
  - логин и пароль,
  - при необходимости — `spring.jpa.hibernate.ddl-auto=validate` или `none`, чтобы структуру править только миграциями, а не Hibernate.
- Проверить, что приложение стартует и подключается к БД (запуск `mvn spring-boot:run` или из IDE).

**Результат:** приложение поднимается, БД доступна, в логах нет ошибок подключения.

## 4. Entity-классы (JPA)

- Для каждой таблицы создать класс-сущность в пакете `entity` (или `domain`):
  - `User`
  - `Company`
  - `MoySkladIntegration`
  - `TelegramIntegration`
  - `TopNReport`
  - `TopNEntry`
- В сущностях:
  - поля — в соответствии с колонками таблиц;
  - связи: `@ManyToOne` / `@OneToMany` там, где есть внешние ключи (например, Company → User, TopNReport → Company, TopNEntry → TopNReport);
  - при необходимости `@CreatedDate` для `created_at`.
- Проверить, что имена полей и таблиц совпадают со схемой (или заданы через `@Table`, `@Column`).

**Результат:** все таблицы отражены в Entity, приложение по-прежнему стартует (если используешь `ddl-auto=validate`, Hibernate проверит соответствие схеме).

---

## 5. Репозитории (Spring Data JPA)

- Создать интерфейсы-репозитории в пакете `repository`:
  - `UserRepository` — `JpaRepository<User, Long>`, при необходимости `Optional<User> findByEmail(String email)`.
  - `CompanyRepository` — `JpaRepository<Company, Long>`, при необходимости `List<Company> findByUserId(Long userId)`.
  - `MoySkladIntegrationRepository` — `JpaRepository<MoySkladIntegration, Long>`, при необходимости `Optional<...> findByCompanyId(Long companyId)`.
  - `TelegramIntegrationRepository` — по аналогии.
  - `TopNReportRepository` — при необходимости `List<TopNReport> findByCompanyIdAndStatus(...)`.
  - `TopNEntryRepository` — при необходимости `List<TopNEntry> findByReportId(Long reportId)`.
- Не писать лишней логики в репозиториях на первой неделе — только то, что нужно для доступа по id и по связям.

**Результат:** все репозитории объявлены, приложение стартует, бины репозиториев создаются.

---

## 6. DTO на первой неделе не нужны

- **DTO (Data Transfer Object)** — классы, в которых бэкенд отдаёт и принимает данные по REST (контракт с фронтом). Они появятся на **неделе 2**, когда будешь определять контракт API и реализовывать его: контроллеры будут принимать/возвращать DTO, а сервис — переводить Entity ↔ DTO.
- На неделе 1 работа только с **Entity и репозиториями**; DTO можно не писать.

---

## 7. Накат структуры при старте (Liquibase)

- Убедиться, что в конфиге указан `spring.liquibase.change-log` (см. п. 2).
- При первом запуске приложения Liquibase выполнит чейнджлоги — таблицы создадутся.
- Дальше при каждом старте Liquibase проверит, какие изменения уже применены (по своей служебной таблице), и накатит только новые.

**Результат:** при первом запуске в БД появляются все таблицы; при следующих изменениях схемы добавляешь новые чейнджлоги.

## 8. Проверка и тесты

- Вручную: открыть БД (DBeaver, pgAdmin, psql), убедиться, что таблицы созданы, колонки и типы совпадают с ожиданием.
- Написать простые unit-тесты под репозитории (например, с H2 in-memory):
  - сохранить сущность (User, Company, …);
  - найти по id;
  - при наличии — найти по email / по companyId и т.д.
- Убедиться, что все тесты проходят.

**Результат:** структура БД накатывается при старте приложения; репозитории проходят unit-тесты; к концу недели 1 есть работающая «основа»: БД + Entity + репозитории.

## 9. Контейнеры: PostgreSQL + бэкенд в Docker Compose

- В корне проекта (или в папке `docker` / `deploy`) создать **docker-compose.yml** с двумя сервисами:
  - **postgres:** образ `postgres:15`, переменные `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, порт `5432:5432`, volume для данных.
  - **backend:** сборка из папки бэкенда (Dockerfile: сборка JAR → образ на JRE, `java -jar`). `depends_on: postgres`. Переменные окружения: `SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/<db>`, `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD` (имя хоста БД — **postgres**, не localhost).
- В `application.yml` бэкенда убедиться, что URL/логин/пароль к БД можно переопределить через переменные окружения (чтобы в контейнере подключаться к `postgres:5432`).
- Создать **Dockerfile** для бэкенда (многоэтапная сборка: Maven/Gradle → JAR, затем образ с JRE и этим JAR).
- Запуск: `docker-compose up -d postgres backend` (или `docker compose up --build`). Проверить: Liquibase накатывает таблицы, приложение отвечает (например, `http://localhost:8080/actuator/health` или простой endpoint).
- В README добавить: как запустить (`docker-compose up --build`), что поднимается (postgres, backend), по каким портам что доступно.

**Результат:** одной командой поднимаются PostgreSQL и бэкенд в контейнерах; БД инициализируется при старте бэка; к концу недели 1 контейнеры уже развёрнуты.
