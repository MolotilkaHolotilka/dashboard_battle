## 1. Механизм работы с токенами пользователей МойСклад

### 1.1 Как устроена аутентификация в API МойСклад

По документации МойСклад JSON API ([dev.moysklad.ru](https://dev.moysklad.ru/doc/api/remap/1.2/)) поддерживаются два способа аутентификации:

1. **Basic Auth** — в заголовке `Authorization` передаётся пара **логин:пароль** в кодировке Base64 (RFC2045-MIME).
2. **Токен доступа** — в заголовке `Authorization` передаётся значение `Bearer <токен>`.

Для интеграции «Битва Дашбордов» предполагается использование **токена доступа**: пользователь создаёт токен в личном кабинете МойСклад и передаёт его в наше приложение. Логин и пароль в нашу систему не передаются, что снижает риски и упрощает отзыв доступа (достаточно отозвать токен в МойСклад).

**Получение нового токена:** выполняется запрос к API с Basic Auth (логин:пароль). В ответ приходит `access_token`. Важно: при генерации нового токена **все ранее выданные токены этого пользователя отзываются**.

**Ограничение прав (scope) для токена:** в документации JSON API и в типовом сценарии создания токена пользователем **нет возможности явно задать scope** (например, «только чтение» или доступ только к отдельным сущностям). Токен наследует права той **учётной записи** МойСклад, под которой он получен. В одном аккаунте МойСклад (одна компания) может быть несколько учётных записей: владелец/администратор и, при необходимости, сотрудники с разными ролями. Ограничить объём доступа по токену можно только так: в том же аккаунте создать **дополнительную** учётную запись (например, «API для Битвы Дашбордов») с урезанными правами (только просмотр отгрузок, сотрудников, отчётов) и выпускать токен от её имени. Тогда при утечке токена злоумышленник получит только эти ограниченные права, а не полные права владельца. Если в МойСклад заходит только владелец и других пользователей в аккаунте нет, отдельную учётную запись создать можно (Настройки → Сотрудники/Пользователи), но это не обязательно — токен в таком случае будет с правами владельца. Параметр `scope` в документации упоминается в контексте дескриптора **Решений МойСклад** (при публикации приложения в каталоге), а не при выдаче пользовательского токена через настройки аккаунта.

## 2. Риски при хранении и использовании токенов (подробно)

| Риск | Описание | Последствия |
|------|----------|-------------|
| **Утечка токена** | Токен хранится в открытом виде в БД, логах, конфигурации или передаётся по незащищённому каналу. | Любой, кто получит токен, получает полный доступ к данным аккаунта МойСклад в рамках прав пользователя, создавшего токен: чтение и изменение товаров, заказов, отгрузок, контрагентов и т.д. |
| **Неограниченный срок жизни** | Токен не отзывается при смене сотрудника, увольнении или смене пароля (если пользователь не сгенерировал новый токен). | Бывший сотрудник или злоумышленник может продолжать пользоваться старым токеном до явного отзыва. |
| **Отсутствие разграничения прав** | Токен обычно выдаётся с теми же правами, что и учётная запись пользователя в МойСклад. | При компрометации токена злоумышленник получает доступ в объёме прав владельца токена (в т.ч. полный доступ при правах администратора). |
| **Хранение в небезопасном месте** | Токены в открытой таблице БД, в переменных окружения в виде plain text на общих серверах, в коде или в незащищённых файлах. | Доступ к серверу, дампу БД или репозиторию даёт доступ ко всем токенам и, как следствие, к аккаунтам пользователей в МойСклад. |
| **Передача по незащищённому каналу** | Ввод токена пользователем по HTTP (без HTTPS) или отправка токена в URL/логах. | Перехват трафика или утечка логов приводит к компрометации токена. |
| **Общий доступ к токенам внутри приложения** | Любой модуль или разработчик может читать все токены из хранилища. | Увеличение площади атаки и риска случайной утечки (логирование, ошибки, сторонние библиотеки). |

## 3. Варианты улучшения безопасности хранения и использования токенов

- **Шифрование токенов в БД (at rest).**  
  Хранить не сам токен, а его зашифрованное значение (AES-256 или аналог). Ключ шифрования хранить отдельно от БД (например, в секретном хранилище / HSM / переменная окружения на защищённом сервере). Расшифровка — только в момент обращения к API МойСклад.

- **Минимизация области видимости.**  
  Поскольку явных scope для токена в МойСклад нет, для аккаунтов с несколькими учётными записями можно рекомендовать создать отдельную учётную запись (например, «API для интеграции») с минимально необходимыми правами и выпускать токен от её имени. Если в аккаунте только владелец — токен будет с его правами; это типичный сценарий для малого бизнеса. Регулярно напоминать о перевыпуске токена при смене ответственных лиц.

- **Защита при передаче.**  
  Только HTTPS для всего веб-интерфейса и API приложения. Токен не передавать в URL, только в теле запроса или заголовке, не логировать.

- **Разделение секретов и кода.**  
  Токены не хранить в коде и не коммитить в репозиторий. Использовать переменные окружения или специализированные хранилища секретов (Vault, секреты в облаке и т.п.) для ключей шифрования и, при необходимости, для самих токенов в production.

- **Ограничение доступа к данным в приложении.**  
  К хранилищу токенов (таблица/сервис) иметь доступ только сервису интеграции с МойСклад; не отдавать токены в ответах API и не выводить в UI (показывать только факт «интеграция настроена» или маскированное значение).

- **Аудит и отзыв.**  
  Логировать факты использования токена (без самого токена): какой пользователь, когда, какой запрос. Реализовать в приложении возможность «отвязать» интеграцию (удалить токен из БД); при необходимости — инструкция пользователю по перевыпуску токена в МойСклад.

- **Для мультитенантного режима.**  
  Токены привязаны к учётной записи пользователя в нашем приложении; доступ к токену только у сервиса, обрабатывающего запросы этого пользователя (по сессии/роли).

## 4. Простой и удобный путь: от пользователя к продукту

Цель — минимизировать шаги и не требовать от пользователя технических действий сверх необходимого.

**Что нужно от пользователя для работы с МойСклад:**

1. Иметь аккаунт МойСклад (логин и пароль).
2. Создать токен доступа (один раз) в настройках МойСклад и скопировать его.
3. Зарегистрироваться/войти в наше приложение (веб).
4. В разделе «Интеграции» или «Настройки» указать, что подключается МойСклад, вставить токен и сохранить.

**Что делает продукт:**

- Принимает токен только по HTTPS.
- Сохраняет токен в зашифрованном виде (см. выше).
- Не показывает токен целиком в интерфейсе после сохранения.
- Даёт пользователю возможность «отвязать» интеграцию (удалить сохранённый токен).
- В инструкции и в UI пошагово описать: где в МойСклад взять токен (Настройки → Интеграции / API → Токены доступа), что вставить в форму и что произойдёт при сохранении.

Такой путь не требует от пользователя установки ПО, работы с API вручную или передачи логина/пароля — только регистрация в приложении и вставка одного токена.

## 5. Что пользователь делает, чтобы получить токен МойСклад (полный список действий)

1. Открыть сайт [МойСклад](https://www.moysklad.ru/) и при необходимости зарегистрировать аккаунт (email, пароль, название компании и т.д.).
2. Войти в личный кабинет (логин и пароль).
3. Перейти в раздел **Настройки** (шестерёнка в интерфейсе).
4. Найти подраздел, связанный с **интеграциями** или **API** (например, «Настройки» → «Интеграции» или «Токены доступа» / «Доступ к API» — точное название уточняется по актуальному интерфейсу МойСклад).
5. Создать новый токен доступа (кнопка «Создать токен» или аналог).
6. При необходимости задать название/назначение токена (например, «Битва Дашбордов»).
7. Скопировать выданный токен и сохранить его в надёжном месте (токен обычно показывается один раз; при повторной генерации старые токены отзываются).

После этого пользователь может передать этот токен в наше приложение (см. раздел 6).

## 6. Что пользователь делает, чтобы передать токен в Продукт (полный список действий)

1. Убедиться, что приложение «Битва Дашбордов» доступно (веб-приложение по ссылке или развёрнуто у заказчика).
2. Открыть в браузере адрес приложения.
3. Зарегистрироваться (если ещё не зарегистрирован): указать email, пароль и прочие обязательные данные по форме регистрации.
4. Войти в личный кабинет приложения (логин/email и пароль).
5. Перейти в раздел настройки интеграций (например, «Настройки», «Интеграции», «Подключить МойСклад»).
6. Выбрать подключение МойСклад и в поле ввода вставить скопированный ранее токен доступа.
7. Нажать «Сохранить» / «Подключить».
8. При успешной проверке (например, тестовый запрос к API МойСклад) увидеть подтверждение, что интеграция активна. При ошибке — сообщение об ошибке и возможность исправить токен.

С этого момента приложение может от имени этого пользователя запрашивать данные из МойСклад в рамках выданного токена. Установка дополнительного ПО на компьютер пользователя не требуется.

## 7. Частота запросов к API МойСклад

### 7.1 Ограничения по документации

- API возвращает **429 (Too Many Requests)** при превышении лимита количества запросов.
- В ответах доступны заголовки для контроля лимита:
  - **X-RateLimit-Limit** — допустимое количество запросов в интервале;
  - **X-RateLimit-Remaining** — сколько запросов осталось до 429;
  - **X-Lognex-Retry-After** / **X-Lognex-Reset** — время до сброса ограничения (мс);
  - **X-Lognex-Retry-TimeInterval** — интервал (мс), в течение которого действует лимит.

- Отдельно действует **ограничение на одновременное количество запросов**: при его превышении возвращается 429 с кодом ошибки **1073** («Превышено ограничение на одновременное количество запросов»).

Точные числовые значения лимитов (запросов в минуту/секунду и число параллельных запросов) в приведённом фрагменте документации не указаны; их необходимо уточнять в актуальной документации или у поддержки МойСклад. Для проектирования разумно закладывать консервативную оценку (порядка сотен запросов в минуту на аккаунт и ограничение параллельных запросов).

### 7.2 Где будет «бутылочное горлышко»

- **Формирование ТОП-N:** нужно получить сотрудников, продажи (отгрузки/документы), при необходимости товары и остатки. Для одного аккаунта это может означать несколько десятков запросов (списки с пагинацией, expand, отчёты). При частом пересчёте (например, каждые несколько минут для многих пользователей) суммарное число запросов растёт.
- **Много пользователей (компаний):** у каждого пользователя свой токен и свой аккаунт МойСклад. Один общий запрос «за всех» в API МойСклад **невозможен** — каждый запрос идёт с одним токеном и возвращает данные только этого аккаунта. Поэтому при N компаниях выполняется N групп запросов; общая нагрузка на наше приложение и на соблюдение лимитов увеличивается.
- **Параллельные запросы:** при одновременной отправке многих запросов с одним токеном легко упереться в лимит на одновременные запросы (429, 1073). Бутылочное горлышко — либо лимит на число запросов в единицу времени, либо лимит на параллельность, в зависимости от того, что достигается первым.

Итого: бутылочное горлышко — **лимиты API МойСклад (на частоту и параллельность запросов)** при росте числа пользователей и/или частоты обновления ТОП-N.

### 7.3 Как оптимизировать частоту запросов

- **Кэширование:** сохранять результаты запросов к API (сотрудники, отгрузки, отчёты) на короткий период (например, 5–15 минут). Формирование ТОП-N делать по кэшу; к API обращаться только при истечении TTL или при принудительном обновлении.
- **Сведение запросов:** использовать выборки с фильтрами и полями `limit`, `offset`, `expand` так, чтобы за один запрос получать максимум нужных данных и не делать лишних постраничных обходов там, где достаточно одной страницы.
- **Очередь и троттлинг:** не дергать API по каждому действию пользователя; ставить задачи в очередь и выполнять запросы к одному аккаунту с ограничением частоты (например, не более K запросов в минуту на токен).
- **Пакетная обработка по расписанию:** пересчёт ТОП-N раз в N минут (или раз в час) по расписанию, а не в реальном времени при каждом открытии страницы.
- **Учёт заголовков лимита:** читать `X-RateLimit-Remaining` и `X-Lognex-Retry-After`; при приближении к лимиту замедлять запросы или откладывать их до сброса окна.

### 7.4 Один запрос для нескольких пользователей

**Такой возможности в API МойСклад нет.** Каждый запрос аутентифицируется одним токеном (Bearer или Basic Auth), и ответ содержит данные только того аккаунта МойСклад, которому принадлежит этот токен. Данные других компаний в одном запросе получить нельзя. Поэтому для каждого подключённого пользователя (каждой компании) приложение выполняет свой набор запросов с его токеном. «Один запрос для нескольких пользователей» в смысле «одним вызовом API получить данные нескольких аккаунтов» документация не предусматривает.

## 8. Перспективы реализации «взаимодействия через Telegram» на альтернативных платформах

Ниже — варианты каналов представления результатов (ТОП-N, метрики) и управления, без привязки только к Telegram.

| Платформа / решение | Описание | Плюсы | Минусы / особенности |
|---------------------|----------|--------|---------------------------|
| **Веб-приложение** | Раздел в нашем приложении (дашборд, отчёты, история публикаций). | Полный контроль над интерфейсом, не зависим от правил мессенджеров, удобно для детальной аналитики и настроек. | Пользователь должен заходить на сайт; нет push-уведомлений «в привычном мессенджере». |
| **Telegram (канал + бот)** | Публикация ТОП-N в канал; при необходимости бот для уведомлений и простых команд. | Привычная среда для многих, мгновенная доставка, возможность обсуждения в комментариях. | Зависимость от правил Telegram, лимиты Bot API. |
| **Другие мессенджеры** | Аналогичная логика для VK, WhatsApp (Business API), и др. | Расширение аудитории. | Отдельная интеграция под каждый мессенджер, разные API и ограничения. |
| **Десктопное / мобильное приложение** | Отдельное приложение (Electron, React Native, Flutter и т.п.) с дашбордом и уведомлениями. | Нативный опыт, уведомления на устройстве. | Высокие затраты на разработку и поддержку нескольких платформ. |
| **Решения МойСклад** | Модуль в каталоге [Решения МойСклад](https://www.moysklad.ru/reshenija/), работающий внутри экосистемы МойСклад. | Прямая видимость для пользователей МойСклад, возможная интеграция с интерфейсом и правами. | Жёсткие требования и процесс публикации; репрезентация «во внешних системах» (например, Telegram) может потребовать отдельного канала вне Решений. |
| **Email-рассылка** | Отправка ТОП-N на email по расписанию. | Не требует установки мессенджеров, привычно для корпоративной среды. | Низкая «живость» по сравнению с мессенджерами, риск попадания в спам. |

**Вывод:** для MVP целесообразно оставить фокус на **веб-приложении** (управление интеграциями, формирование и одобрение ТОП-N) и **Telegram-канале** (публикация одобренных результатов). Остальные платформы можно подключать по мере потребности и ресурсов; веб + Telegram покрывают сценарий «настроил — получил результат в канале» при разумных трудозатратах.

## 9. Постановка главной цели проекта

### 9.1 Цель

Реализовать проект **«Разработка программного комплекса „Битва Дашбордов" для анализа производительности персонала, занятого в продажах, на предприятиях под управлением CRM „МойСклад" и репрезентации статистики во внешних информационных системах»**.

### 9.2 Этапы и сроки

| Неделя | Этап | Содержание | Критерии завершения |
|--------|------|------------|---------------------|
| 1 | База данных + контейнеры | Проектирование и реализация структуры БД (Liquibase, Entity, репозитории). Docker Compose: PostgreSQL + бэкенд (Spring Boot) в контейнерах. | Структура БД накатывается при старте; репозитории проходят unit-тесты; `docker-compose up` поднимает postgres и backend. |
| 2 | Фронтенд и контракт с бэком | Каркас React (экраны-заглушки). Определение контракта API (OpenAPI). Реализация на бэке (REST, DTO). Фронт в docker-compose. | Фронт отображает экраны; контракт зафиксирован; бэк отдаёт/принимает по контракту; `docker-compose up` поднимает postgres, backend и frontend. |

);

CREATE TABLE top_n_entries (
    id              BIGSERIAL PRIMARY KEY,
    report_id       BIGINT REFERENCES top_n_reports(id) ON DELETE CASCADE,
    employee_name   VARCHAR(255),
    employee_id_ms  VARCHAR(100),  -- ID сотрудника в МойСклад
    revenue         NUMERIC(15,2),
    margin          NUMERIC(15,2),
    favorite_product VARCHAR(255),
    rank            INT
);

## 12. Внешние системы и методы API

### 12.1 МойСклад JSON API

Документация: https://dev.moysklad.ru/doc/api/remap/1.2/

**Методы, которые будет использовать приложение:**

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| GET | `/entity/employee` | Список сотрудников |
| GET | `/entity/demand` | Список отгрузок (продаж) |
| GET | `/entity/retaildemand` | Розничные продажи |
| GET | `/report/profit/byemployee` | Прибыль по сотрудникам (если доступно) |
| GET | `/entity/product` | Товары (для «любимого» товара) |
| GET | `/security/token` | Проверка/получение токена (при необходимости) |

Параметры: `filter`, `limit`, `offset`, `expand`, `order`.

### 12.2 Telegram Bot API

Документация: https://core.telegram.org/bots/api

| Метод | Назначение |
|-------|------------|
| `getMe` | Проверка токена бота |
| `sendMessage` | Отправка сообщения в канал |
| `sendPhoto` / `sendDocument` | Отправка изображения/файла (если нужно) |

**Логика:**
- Бот добавлен администратором в канал.
- Приложение хранит `bot_token` и `channel_chat_id`.
- При публикации ТОП-N вызывается `sendMessage` с форматированным текстом.
