# Неделя 2 — Фронтенд и контракт с бэком: развёрнутый план

Задача недели: сделать каркас фронтенда (React), определить, что фронт отправляет и получает от бэкенда (контракт API), и реализовать этот контракт на бэке (REST, DTO). Контейнеры (postgres + backend) уже поднимаются с недели 1; на 2-й неделе добавляется фронт в compose и «живое» взаимодействие фронт ↔ бэк.

---

## 1. Каркас фронтенда (React)

- Создать проект фронтенда в папке `frontend` (Vite + React или Create React App).
- Подключить Redux и Redux Saga (или иной способ работы с состоянием и асинхронными вызовами к API).
- Сделать базовую маршрутизацию (React Router): заглушки страниц под основные экраны:
  - вход / регистрация (пока без логики, только разметка);
  - настройки интеграций (МойСклад, Telegram) — заглушка;
  - список отчётов ТОП-N и просмотр одного отчёта — заглушка.
- Настроить обращение к бэкенду: базовый URL API вынести в конфиг/переменную окружения (например, `VITE_API_URL=http://localhost:8080` или через proxy в dev-сервере), чтобы фронт мог слать запросы на бэк.

**Результат:** приложение React с несколькими экранами-заглушками и настроенным базовым URL к API.

---

## 2. Определение контракта API

- Зафиксировать список запросов и ответов между фронтом и бэком. Минимум для MVP:
  - **Пользователь:** регистрация (POST), вход (POST), получение текущего пользователя (GET).
  - **Компании:** список компаний пользователя (GET), создание компании (POST).
  - **Интеграции МойСклад / Telegram:** CRUD (создать, прочитать, обновить, удалить) — какие поля передаются и возвращаются (без секретов в ответах).
  - **ТОП-N:** создание отчёта (POST с периодом и company_id), список отчётов (GET), получение одного отчёта со строками (GET), одобрение/отклонение (PATCH), публикация (POST).
- Оформить контракт в виде **OpenAPI (Swagger)** спецификации: пути, методы, request/response body (JSON-схема). Файл можно хранить в репозитории (например, `openapi.yaml` или в папке `docs`). По нему потом генерировать клиент на фронте или просто держать как источник правды.
- Согласовать с собой (или с куратором): что именно фронт будет показывать на каждом экране и какие вызовы API для этого нужны.

**Результат:** документ/файл с контрактом API (OpenAPI); понятно, какие endpoints есть и какие DTO уходят и приходят.

---

## 3. Реализация контракта на бэкенде (REST, DTO)

- Для каждого ресурса из контракта:
  - создать **DTO** (request/response): например, `UserRegisterDto`, `UserDto`, `CompanyDto`, `TopNReportDto`, `TopNEntryDto` и т.д.;
  - создать **REST-контроллер** (или методы в существующих контроллерах): принять DTO, вызвать сервис, вернуть DTO;
  - в **сервисе** — минимальная логика: обращение к репозиториям, преобразование Entity → DTO и обратно (маппинг вручную или через MapStruct).
- Не тянуть в ответы чувствительные данные (токены МойСклад/Telegram не возвращать, только факт «интеграция настроена» или id).
- По возможности подключить **Swagger UI** (springdoc-openapi или аналог), чтобы по адресу типа `http://localhost:8080/swagger-ui.html` была документация и возможность «потыкать» API. Сверить с описанным контрактом.

**Результат:** бэкенд отдаёт и принимает данные по контракту; можно вызывать API с Postman/curl или со Swagger UI; данные сохраняются в БД через репозитории.

---

## 4. Фронтенд вызывает бэкенд

- На каждом экране (или в сагах/сервисах) реализовать вызовы к API по контракту: например, при загрузке страницы «Мои компании» — GET `/api/companies`, при создании отчёта ТОП-N — POST с периодом и company_id.
- Обработать ответы: успех (обновить состояние, показать данные) и ошибки (показать сообщение пользователю или залогировать).
- Проверить, что при запущенном бэке фронт реально получает данные (например, список компаний или пустой массив) и может отправить данные (создание компании, создание отчёта). CORS на бэке должен разрешать запросы с адреса фронта (например, `http://localhost:3000`).

**Результат:** фронт и бэк работают вместе по контракту; на экранах отображаются данные с бэка (или заглушки, если данных ещё нет).

---

## 5. Фронтенд в Docker Compose

- Добавить в существующий **docker-compose.yml** сервис **frontend**:
  - Dockerfile в папке `frontend`: либо dev-режим (Node, `npm run dev`), либо сборка статики + nginx.
  - Порт (например, `3000:3000` или `80:80`).
  - Переменная окружения с URL бэкенда: в контейнере фронт должен обращаться к бэку по имени сервиса, например `http://backend:8080` (если запросы идут с сервера) или оставить `http://localhost:8080` для запросов из браузера (браузер живёт на хосте).
- После `docker-compose up --build` проверять: открывается ли фронт в браузере, уходят ли запросы на бэк (по адресу хоста, т.е. `localhost:8080`, если фронт только отдаёт статику, а запросы из браузера).

**Результат:** `docker-compose up` поднимает postgres, backend и frontend; фронт доступен по своему порту и может общаться с бэком.

---

## 6. Проверка и тесты

- Вручную пройти основные сценарии: открыть фронт → залогиниться (или зарегистрироваться) → открыть компании → создать отчёт ТОП-N (заглушка или с мок-данными).
- При необходимости добавить простые unit-тесты на бэкенде (сервисы, маппинг Entity ↔ DTO), чтобы не сломать контракт при изменениях.

**Результат:** уверенность, что контракт соблюдается и фронт с бэком работают вместе.
