**Тема:** Разработка комплекса «Битва Дашбордов» для анализа производительности продавцов по данным МойСклад и публикации результатов во внешние системы.

---

## 1-й предварительный этап

### 1.1. Основные работы

1. Проведение анализа предметной области: изучение возможностей CRM «МойСклад» (учёт продаж, сотрудники, отгрузки, отчёты), определение требований к формированию рейтингов продавцов (ТОП-N) и к репрезентации результатов во внешних системах.

2. Изучение документации и ограничений API МойСклад (JSON API): аутентификация (токены доступа), лимиты запросов, доступные сущности и отчёты (сотрудники, отгрузки, прибыль по сотрудникам и т.п.).

3. Изучение Telegram Bot API: возможности публикации сообщений в канал, ограничения и требования к интеграции.

4. Разработка концептуальной архитектуры программного комплекса: взаимодействие фронтенда (React, Node.js), бэкенда (Spring Boot), СУБД (PostgreSQL), внешних систем (МойСклад, Telegram); определение принципов хранения и защиты пользовательских данных (токены, шифрование).

5. Проектирование базы данных: определение сущностей (пользователи, компании, интеграции с МойСклад и Telegram, отчёты ТОП-N и их строки), связей между ними и структуры таблиц; выбор средства миграций (Liquibase).

6. Анализ рисков при работе с токенами пользователей МойСклад и способов повышения безопасности (хранение, передача, минимизация прав).

7. Определение контракта API между фронтендом и бэкендом (список операций, форматы запросов и ответов) и способа его фиксации (OpenAPI/Swagger).

### 1.2. Дополнительные работы

- Проектирование сценариев взаимодействия пользователя с системой: получение токена МойСклад, передача токена в приложение, настройка интеграций, создание и одобрение отчёта ТОП-N, публикация в Telegram-канал.

- Анализ альтернативных платформ для репрезентации результатов (веб-приложение, другие мессенджеры, Решения МойСклад, email) и обоснование выбора (веб + Telegram-канал для MVP).

- Подготовка диаграмм и описаний: схема БД, схема взаимодействия компонентов (фронт — бэк — БД — МойСклад, Telegram).

---

## 2-й предварительный этап

### 2.1. Основные работы

1. Разработка технического задания на программный комплекс (в объёме, предусмотренном требованиями к практике и ГОСТ 7.32-2017 при необходимости).

2. Реализация структуры базы данных: создание миграций Liquibase, сущностей JPA (Entity) и репозиториев (Spring Data JPA) для всех спроектированных таблиц.

3. Настройка и развёртывание окружения в контейнерах: Docker Compose (PostgreSQL, бэкенд Spring Boot, при необходимости фронтенд); обеспечение инициализации БД при старте приложения.

4. Реализация бэкенда: REST API по контракту, сервисный слой, преобразование Entity ↔ DTO; интеграционные модули для взаимодействия с API МойСклад и Telegram Bot API (в т.ч. с возможностью работы на моках для демонстрации).

5. Реализация фронтенда: каркас приложения на React (Redux, Redux Saga), экраны для управления интеграциями, создания и просмотра отчётов ТОП-N, публикации в Telegram; обеспечение взаимодействия с бэкендом по зафиксированному контракту.

6. Реализация бизнес-логики формирования рейтинга ТОП-N на основе данных из МойСклад (выборка сотрудников, отгрузок/продаж, расчёт метрик, сортировка и формирование отчёта).

7. Подготовка документации: README с инструкцией по запуску (Docker, команды, порты), описание API (Swagger/OpenAPI), руководство пользователя.

8. Проведение и описание тестирования: разработка и прогон unit-тестов (сервисы, репозитории, маппинг), тест-кейсы для ручной проверки; при необходимости — UI-тесты (Selenium); соблюдение ограничений при работе с внешними API (МойСклад, Telegram). Фиксация результатов тестирования.

9. Размещение программного кода и документации в системе контроля версий (Git), ведение истории изменений по этапам практики.

10. Подготовка отчёта по производственной практике.

### 2.2. Дополнительные работы

- Настройка CI/CD (например, GitHub Actions или Coolify) для сборки и развёртывания проекта.

- Разработка стратегии кэширования и ограничения частоты запросов к API МойСклад для соблюдения лимитов и снижения нагрузки.

- Поэтапное размещение кода и документации в системе контроля версий (Git), ведение истории изменений по этапам практики.
