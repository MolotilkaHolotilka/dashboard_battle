Соответствие пунктов задания и файлов/разделов в проекте.

## 1. Скрипт или логика автоматизации запуска бэк-приложения

### 1.1 Организация работы СУБД PostgreSQL в контейнере

**Смотреть:** корень проекта, файл **`docker-compose.yml`**.

- Сервис **`postgres`**: образ `postgres:15`, переменные `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, порт `5432:5432`, volume `postgres_data` для данных, healthcheck.

---

### 1.2 Организация работы базового Spring Boot приложения в контейнере

**Смотреть:**

- **`docker-compose.yml`** — сервис **`backend`**: `build: ./backend`, порт `8080:8080`, переменные окружения для БД.
- **`backend/Dockerfile`** — двухэтапная сборка: этап 1 — Maven и сборка JAR, этап 2 — образ на JRE, запуск `java -jar app.jar`.

#### 1.2.1 Связь Spring Boot с контейнером, где развёрнута СУБД PostgreSQL

**Смотреть:**

- **`docker-compose.yml`** — у сервиса `backend`: `depends_on: postgres` (с условием `service_healthy`), переменные `SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/dashboard_battle`, `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD`. Хост БД в контейнере — имя сервиса **`postgres`**.
- **`backend/src/main/resources/application.yml`** — секция `spring.datasource`: URL/логин/пароль берутся из переменных окружения (`${SPRING_DATASOURCE_URL:...}` и т.д.), чтобы в Docker подставлялись значения из compose.

#### 1.2.2 Развёртывание БД при запуске Spring Boot (без повторного выполнения скриптов)

**Смотреть:**

- **`backend/src/main/resources/application.yml`** — `spring.liquibase.change-log: classpath:db/changelog/db.changelog-master.xml`, `spring.jpa.hibernate.ddl-auto: validate`.
- **`backend/src/main/resources/db/changelog/db.changelog-master.xml`** — главный чейнджлог Liquibase, подключает файлы миграций.
- **`backend/src/main/resources/db/changelog/changelog/001-create-tables.xml`** — изменения схемы (createTable для users, companies, moysklad_integrations, telegram_integrations, top_n_reports, top_n_entries). Liquibase записывает выполненные changeset’ы в таблицу `databasechangelog` и не выполняет их повторно.

## 3. Демонстрация успешного развёртывания БД

**Смотреть:** **`docs/LOCAL_RUN.md`** — раздел «Доступ к СУБД в контейнере»:

- Команда для входа в psql внутри контейнера:  
  `docker exec -it dashboard-battle-postgres psql -U postgres -d dashboard_battle`
- Команды для просмотра структуры в psql: `\dt`, `\d имя_таблицы`.
- Параметры для подключения с локальной машины (DBeaver, pgAdmin): хост `localhost`, порт `5432`, база `dashboard_battle`, пользователь и пароль `postgres`.

Имя контейнера может отличаться (например, `my_sklad_project-postgres-1`). Актуальное имя: `docker ps` (колонка NAMES).

## 5. Заготовка главного сервиса с инжектами репозиториев

**Смотреть:** **`backend/src/main/java/ru/dashboardbattle/service/DashboardBattleService.java`** — класс с `@Service`, конструктор с параметрами (все шесть репозиториев), геттеры для репозиториев. Логики пока нет, только инжект.

---

## 6. Тесты

**Смотреть:** **`backend/src/test/java/ru/dashboardbattle/DashboardBattleApplicationTests.java`**:

- **Проверка инжекта зависимостей в главный сервис:** тест `contextLoads()` — поднимается контекст (`@SpringBootTest`), проверяется, что бин `DashboardBattleService` и все его репозитории не null.
- **Проверка записи и удаления через репозитории:** тесты `userRepository_saveFindAndDelete()` и `companyRepository_saveFindAndDelete()` — сохраняют сущность в БД, находят по id, удаляют, проверяют отсутствие записи. Используется H2 in-memory (настройки в **`backend/src/test/resources/application.yml`**).

Запуск тестов: из папки **`backend`** команда **`mvn test`**.
