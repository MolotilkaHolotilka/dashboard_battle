# Неделя 4 — что сделано, что нет, как показывать наставнику

Краткий отчёт и инструкция для демонстрации и понимания проекта.

---

## 1. Что сделано

### 1.1 Основа приложения (бэкенд)

- **Spring Boot 3.2**, Java 17, Maven.
- Проект лежит в папке **`backend/`**. Всё приложение — один модуль (нет фронтенда на этой неделе).

### 1.2 База данных

- **PostgreSQL** — основная БД (в Docker или локально).
- **Liquibase** — миграции: при первом запуске приложения автоматически создаются таблицы (`users`, `companies`, `moysklad_integrations`, `telegram_integrations`, `top_n_reports`, `top_n_entries`). Повторный запуск не создаёт таблицы заново (идемпотентность).
- Схема БД описана в **[docs/DB_SCHEMA.md](DB_SCHEMA.md)** — таблицы, поля, связи, связь с бизнес-сущностями.

### 1.3 Сущности и репозитории

- **JPA-сущности** в `backend/src/main/java/ru/dashboardbattle/entity/`: User, Company, MoySkladIntegration, TelegramIntegration, TopNReport, TopNEntry. Они соответствуют таблицам БД.
- **Репозитории** в `backend/src/main/java/ru/dashboardbattle/repository/`: по одному на каждую сущность (UserRepository, CompanyRepository и т.д.). Через них приложение читает и пишет в БД.
- **Главный сервис** `DashboardBattleService` в пакете `service`: пока только «заготовка» — в него внедрены все репозитории. Бизнес-логики (регистрация, ТОП-N, публикация) на неделе 4 нет — это неделя 5.

### 1.4 Тесты

- **Unit-тесты** в `backend/src/test/java/ru/dashboardbattle/DashboardBattleApplicationTests.java`:
  - поднимается контекст Spring, в нём есть главный сервис и все репозитории;
  - проверяется запись и удаление в БД через репозитории (User, Company).  
- Тесты используют **H2** (встроенная БД в памяти), не требуют запущенного PostgreSQL.
- Запуск: из папки `backend` команда **`mvn test`**.

### 1.5 Запуск и инфраструктура

- **Два способа запуска:**
  1. **Локально (Maven):** из папки `backend` — `mvn spring-boot:run`. Нужна установленная БД PostgreSQL с базой `dashboard_battle` и пользователем `postgres`/`postgres`, либо запуск с профилем `no-db` (тогда используется H2 в памяти, без Liquibase).
  2. **В контейнерах:** из корня проекта — `docker compose up -d`. Поднимаются контейнеры PostgreSQL и бэкенд; Liquibase при старте бэкенда создаёт таблицы в БД в контейнере.
- **Скрипты:** `scripts/start.bat` (Windows) и `scripts/start.sh` (Linux/macOS) — проверяют наличие Docker и запускают `docker compose up -d`.
- **Docker:** в корне проекта есть `docker-compose.yml` (сервисы postgres и backend) и `backend/Dockerfile` (сборка JAR и образ на JRE).

### 1.6 Документация

- **README.md** — описание проекта, раздел «Запуск проекта» (требования, Docker, локальный запуск, порты).
- **docs/DB_SCHEMA.md** — схема БД и связь таблиц с бизнес-сущностями.
- **docs/LOCAL_RUN.md** — как подключиться к БД в контейнере (psql, DBeaver/pgAdmin).
- **docs/SCRIPTS.md** — листинг скриптов запуска и пояснения.

## 3. Как показывать наставнику (пошагово)

Ниже — сценарий демонстрации: что запускать, что говорить, что открыть в коде.

### Шаг 1. Показать структуру проекта

- Открыть репозиторий в IDE (корень `my_sklad_project`).
- Показать:
  - **`backend/`** — единственное приложение на этой неделе (Maven, Spring Boot).
  - **`docs/`** — описание схемы БД, инструкции по запуску, скрипты.
  - **`scripts/`** — start.bat, start.sh.
  - В корне — **`docker-compose.yml`**, **`README.md`**.

Кратко сказать: «Сделана основа: бэкенд в `backend/`, БД через Liquibase, сущности и репозитории, один главный сервис-заготовка, тесты, Docker и скрипты запуска».

### Шаг 2. Запустить тесты

- В терминале: перейти в `backend`, выполнить **`mvn test`**.
- Должны пройти все тесты (контекст, запись/удаление через репозитории).
- Сказать: «Тесты проверяют, что Spring поднимается, главный сервис и репозитории есть в контексте, и что через репозитории можно сохранять и удалять данные (на примере User и Company). Тесты идут на H2, PostgreSQL не нужен».

### Шаг 3. Запустить приложение

**Вариант А — с локальным PostgreSQL (если у наставника/тебя установлен Postgres):**

1. Создать базу: `createdb dashboard_battle` (или через pgAdmin/DBeaver).
2. В терминале: `cd backend`, затем  
   **`mvn spring-boot:run`**  
   (логин/пароль по умолчанию — postgres/postgres, см. `application.yml`).
3. В логах должно быть: Liquibase выполнил миграции, приложение стартовало, Tomcat на порту 8080.

**Вариант Б — без PostgreSQL (только показать, что приложение поднимается):**

- В терминале: `cd backend`, затем  
  **`mvn spring-boot:run "-Dspring-boot.run.profiles=no-db"`**  
  (в кавычках, чтобы профиль передался Maven).
- Приложение поднимется на H2 в памяти, Liquibase отключён, таблицы создаёт Hibernate.

После старта открыть в браузере: **http://localhost:8080/actuator/health**  
Показать ответ: `{"status":"UP"}` — «Приложение живое, health endpoint отвечает».

### Шаг 4. Показать базу данных (если запускали с PostgreSQL)

- Если запуск был через **Docker** (`docker compose up -d`):  
  Подключиться к БД в контейнере:  
  **`docker exec -it dashboard-battle-postgres psql -U postgres -d dashboard_battle`**  
  В psql: **`\dt`** — список таблиц; **`\d users`** — структура таблицы.
- Если запуск был **локально** — подключиться к той же БД через DBeaver/pgAdmin (localhost:5432, база `dashboard_battle`, пользователь postgres).
- Сказать: «Таблицы созданы Liquibase при первом старте приложения. Схема описана в docs/DB_SCHEMA.md».

### Шаг 5. Показать код (как что связано)

- **Точка входа:** `backend/src/main/java/ru/dashboardbattle/DashboardBattleApplication.java` — класс с `@SpringBootApplication`, запуск приложения.
- **Конфиг:** `backend/src/main/resources/application.yml` — порт 8080, параметры БД (и переменные окружения для Docker), включён Liquibase, профиль `no-db` для запуска без Postgres.
- **Миграции:** `backend/src/main/resources/db/changelog/` — `db.changelog-master.xml` подключает `changelog/001-create-tables.xml`; там создание таблиц. «При старте Liquibase смотрит, какие changeset’ы уже выполнены, и выполняет только новые».
- **Сущности:** пакет `entity` — классы User, Company, MoySkladIntegration, TelegramIntegration, TopNReport, TopNEntry. «Каждая сущность — одна таблица, поля — колонки, связи — внешние ключи».
- **Репозитории:** пакет `repository` — интерфейсы, наследующие JpaRepository. «Spring Data сам реализует методы save, findById, delete и методы вида findByUser_Id — по ним генерируется SQL».
- **Сервис:** `service/DashboardBattleService.java` — в конструктор внедрены все репозитории. «Пока это заготовка: на неделе 5 сюда добавим логику регистрации, ТОП-N, публикации; контроллеры будут вызывать этот сервис».

### Шаг 6. Docker и скрипты (если наставник спрашивает про деплой)

- Показать **`docker-compose.yml`** в корне: два сервиса — postgres и backend; backend зависит от postgres, переменные окружения подставляют URL к БД в контейнере.
- Показать **`backend/Dockerfile`**: два этапа — сборка JAR (Maven), затем образ на JRE с этим JAR.
- Сказать: «Из корня команда `docker compose up -d` поднимает и БД, и бэкенд. Скрипты start.bat и start.sh делают то же самое и проверяют, что установлены Docker и Docker Compose».
- Листинг скриптов и пояснения — в **docs/SCRIPTS.md**.

## 4. Как это всё работает (для твоего понимания)

### Цепочка при запуске приложения

1. Запускается **DashboardBattleApplication** (метод `main`).
2. Spring поднимает контекст: читает **application.yml**, создаёт бин **DataSource** (подключение к БД).
3. **Liquibase** (если не отключён профилем) сравнивает папку `db/changelog/` с таблицей `databasechangelog` в БД и выполняет ещё не выполненные changeset’ы — создаёт таблицы.
4. **JPA/Hibernate** при `ddl-auto=validate` сверяет сущности (entity) со схемой БД; если не совпадает — ошибка при старте.
5. Spring создаёт бины **репозиториев** (по интерфейсам в пакете `repository`) и **DashboardBattleService** (в него инжектятся репозитории).
6. Поднимается **Tomcat** на порту 8080; **Actuator** отдаёт `/actuator/health` и др.

Сейчас приложение **ничего не отдаёт по REST**, кроме Actuator: контроллеров с нашим API ещё нет. Поэтому «как это работает» для пользователя: только health-check и, при запуске с БД, факт что таблицы созданы и репозитории могут с ними работать (это и проверяют тесты).

### Как связаны БД, сущности и репозитории

- **Таблица в БД** (например, `users`) ↔ **Класс-сущность** (`User.java`) — имена полей и таблиц совпадают (или заданы через `@Table`, `@Column`).
- **Репозиторий** (`UserRepository`) — интерфейс без реализации. Spring Data по имени метода (например, `findByEmail`) генерирует запрос к БД и возвращает сущность или список.
- **Сервис** получает репозитории через конструктор и в будущем будет вызывать их для регистрации, сохранения отчётов, публикации и т.д. Контроллеры (неделя 5) будут принимать HTTP-запросы и вызывать методы сервиса.

### Зачем профиль no-db

- По умолчанию приложение ждёт **PostgreSQL** на localhost:5432, база `dashboard_battle`. Если БД нет или пароль другой — старт упадёт.
- Профиль **no-db** переключает на **H2** в памяти и отключает Liquibase. Нужен только чтобы «просто поднять приложение» без настройки Postgres (например, на машине наставника или для быстрой проверки). Для полноценной работы с миграциями и реалистичной БД лучше запускать с PostgreSQL (локально или в Docker).
