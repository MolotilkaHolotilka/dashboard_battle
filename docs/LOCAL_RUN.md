# Локальный запуск и доступ к БД

## Краткий экскурс: как открыть приложение и что в контейнерах

### Нужны ли на компьютере PostgreSQL и Java?

**Если запускаешь через Docker — не нужны.** Достаточно установленного **Docker Desktop**. Внутри контейнеров уже есть и PostgreSQL, и среда выполнения Java (JRE). На твоей машине ничего из этого ставить не обязательно.

**Если запускаешь без Docker** (команда `mvn spring-boot:run` из папки `backend`) — тогда нужны: установленные **JDK 17+** и **Maven**, а также **PostgreSQL** с созданной базой `dashboard_battle` (или профиль `no-db` для запуска на встроенной H2 без Postgres).

### Как открыть приложение (через Docker)

1. Запусти **Docker Desktop**.
2. В терминале перейди в корень проекта и выполни:
   ```powershell
   docker compose up -d
   ```
   или запусти скрипт `scripts\start.bat`.
3. Подожди, пока образы скачаются/соберутся (первый раз может занять несколько минут).
4. Открой в браузере: **http://localhost:8080/actuator/health** — должен вернуться JSON со статусом `UP`.

На этом этапе у приложения нет веб-страниц с формами — только технический endpoint здоровья. Рабочий API и интерфейс появятся на следующих неделях.

### Что «зашито» в контейнеры

| Контейнер | Что внутри | Зачем |
|-----------|------------|--------|
| **postgres** | Образ PostgreSQL 15, база `dashboard_battle`, пользователь/пароль `postgres`. Данные БД хранятся в volume (не пропадают при перезапуске контейнера). | Только СУБД. Ни Java, ни твоего кода там нет. |
| **backend** | Образ собран из нашего кода: этап 1 — Maven собирает JAR, этап 2 — в образ кладётся только JRE (Java Runtime) и этот JAR. При старте контейнера запускается `java -jar app.jar`. Внутри контейнера нет ни Maven, ни исходников — только скомпилированное приложение и JRE. | Только наше Spring Boot приложение. Подключается к БД по имени хоста `postgres` внутри общей сети Docker. |

Итого: **два контейнера** — один с БД, другой с приложением. Postgres и Java на твоём компьютере не требуются, если пользуешься Docker.

## Доступ к СУБД в контейнере

После запуска `docker compose up -d` контейнер PostgreSQL доступен по имени сервиса `postgres` из других контейнеров и по порту **5432** с хоста.

### Подключение через psql внутри контейнера

Имя контейнера по умолчанию: `dashboard-battle-postgres`. База: `dashboard_battle`, пользователь и пароль: `postgres`.

```bash
docker exec -it dashboard-battle-postgres psql -U postgres -d dashboard_battle
```

В интерактивной сессии psql:

- Список таблиц: `\dt`
- Описание таблицы: `\d имя_таблицы` (например, `\d users`)
- Выход: `\q`

### Подключение с хоста (DBeaver, pgAdmin, другой клиент)

| Параметр   | Значение            |
|------------|---------------------|
| Хост       | `localhost`         |
| Порт       | `5432`              |
| База       | `dashboard_battle`  |
| Пользователь | `postgres`        |
| Пароль     | `postgres`          |

После первого успешного старта бэкенда Liquibase создаёт таблицы: `users`, `companies`, `moysklad_integrations`, `telegram_integrations`, `top_n_reports`, `top_n_entries`, а также служебную таблицу `databasechangelog`.
