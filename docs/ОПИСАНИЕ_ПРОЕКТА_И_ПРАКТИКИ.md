## 2. Цель проекта

Реализовать проект в формулировке выше:

- Анализ производительности **персонала по продажам** на основе данных из **МойСклад**.
- Формирование **ТОП-N** (по выручке, приросту, марже и т.п.).
- **Репрезентация статистики** во внешних системах (в MVP — Telegram-канал).

Дополнительно: возможность указывать «любимый» товар лидера по прибыли, мотивация и бенчмаркинг внутри команды.

## 4. Стек (актуальный по итогам Недели 3)

### 4.1 Фронтенд

- **Node.js (Express)** — фронтовое приложение и REST для UI.
- **React (Redux, Redux Saga)** — UI.
- Шаблонизаторы: Mustache, liquidjs, Pug (по необходимости).

### 4.2 Бэкенд

- **Java, Spring Boot (Spring MVC)** — REST API, JSON.
- **Репозитории** для работы с БД, сущности (Entity), DTO.
- **Сервис** — связь контроллера, репозиториев и интеграционных модулей; без тяжёлой логики в контроллере.
- **Интеграционные модули:** МойСклад, Telegram.

### 4.3 Данные и инфраструктура

- **СУБД:** PostgreSQL (для MVP допустим SQLite).
- **Миграции:** Liquibase или набор SQL-скриптов при старте приложения.
- **Nginx, Docker, Docker Compose** — развёртывание (в т.ч. через Coolify).

### 4.4 Тестирование

- **Unit:** логика, сервисы, репозитории, нативные SQL/сложные методы, конвертеры Entity ↔ DTO.
- **UI:** Selenium (имитация пользователя в браузере).
- **Функциональное, чек-листы, нагрузочное** — с соблюдением культуры работы с API (МойСклад не перегружать, не нарушать лимиты Telegram).

## 5. Бизнес-сущности и сценарии

### 5.1 Регистрация и авторизация

- **Пользователь:** личные данные для идентификации; данные для внешних систем (МойСклад, Telegram); авторизационные данные в нашей системе.
- **Запросы:** регистрация, авторизация, проверка статуса сессии.
- **Ответы:** результат регистрации/авторизации, статус сессии.

### 5.2 Управление интеграциями

**МойСклад:**

- Создать интеграцию для пользователя.
- Обновить параметры, удалить, получить данные об интеграции (CRUD).

**Telegram:**

- Аналогичный CRUD для привязки каналов/бота.

### 5.3 Бизнес-запросы к приложению

- Создать **ТОП-N** на основе данных МойСклад → получить DTO с результатами на одобрение.
- **Одобрить** DTO с ТОП-N.
- **Отклонить** DTO (удалить запись или пометить неактивной).
- Получить список доступных Telegram-каналов.
- **Разместить** одобренную DTO с ТОП-N в выбранном Telegram-канале.

### 5.4 Сущность «Компания»

- Создание компании.
- Связь с аккаунтами МойСклад.
- Связь с Telegram-каналами.

### 5.5 Ответы системы

- Регистрация/авторизация и статус сессии.
- Результаты CRUD по интеграциям (МойСклад, Telegram).
- Управление аккаунтами МойСклад и Telegram, компаниями.
- Бизнес-логика ТОП-N: создание, одобрение, отклонение, публикация, отмена публикации.
- **TopNDto** — структура данных для ТОП-N.

## 6. Хранение в БД и слой доступа

- Определить, **какие бизнес-сущности** хранятся в БД.
- Для каждой сущности — **Entity**; где нужно — пара Entity ↔ DTO.
- Для всех Entity — **репозитории** с описанной логикой (SQL/HQL/и т.п.).
- На основе сущностей — **структура БД** (SQL или Liquibase; SQL-представление желательно иметь для схемы).

## 7. Внешние системы и ограничения

### 7.1 МойСклад

- **Документация API:** https://dev.moysklad.ru/
- Работа через **JSON API**, авторизация — токен пользователя (Bearer).
- **Риски:** частота запросов (rate limit), «бутылочное горлышко» при частых обновлениях; безопасное хранение и использование токенов пользователей.
- Получение токена: регистрация в МойСклад → Настройки → Интеграции / API → создание токена. Передача в продукт — через веб-интерфейс приложения (настройка интеграции).

### 7.2 Telegram

- **Управленческая часть:** бот (подъём и описание логики) — в текущей реализации может быть упрощён или отложен; основной сценарий — веб-интерфейс + канал.
- **Репрезентативная часть:** канал для подписчиков, публикация одобренных ТОП-N.
